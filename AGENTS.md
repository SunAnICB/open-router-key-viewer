# AGENTS.md

## 项目定位

这是一个 Python 桌面项目。

- 依赖管理与命令执行使用 `uv`
- UI 技术栈使用 Py Fluent UI
- 当前目标是维护一个 OpenRouter 配额 / 余额 / 配置管理工具

## 基本原则

- 优先保持实现简单、直接、可运行
- 先保证功能闭环，再做抽象
- 不为了“未来可能需要”提前过度设计
- 改动应尽量小而清晰，避免顺手做无关重构

## 环境与工具

### Python

- 默认使用 Python 3.11+
- 新增代码应补全类型标注
- 除非已有约定不同，统一使用 `pathlib.Path`

### 包与命令

- 使用 `uv add` 管理运行时依赖
- 使用 `uv add --dev` 管理开发依赖
- 使用 `uv run` 执行项目命令，不直接调用 `python`、`pip`

示例：

```bash
uv sync
uv run python -m open_router_key_viewer
./scripts/run.sh
./scripts/release.sh
```

## UI 约定

### Py Fluent UI

- 保持界面结构与业务逻辑分离
- 页面层只负责交互、状态分发与展示
- 新增逻辑时尽量不要再把网络请求、文件读写、配置解析继续堆进 UI 组件
- 主题、样式、配色集中管理，不要把样式常量散落在各页面

### 交互设计

- 默认优先清晰、稳定、低认知负担
- 关键操作要有明确反馈
- 对密钥、配置、网络异常等场景提供可理解的错误提示
- 对通知类能力，优先区分应用内提示、系统通知、Webhook 三种渠道
- UI 文案面向最终用户，避免出现“更接近…”、“设计上考虑…”之类设计过程描述

## 代码风格

- 遵循 PEP 8
- 优先小函数、低耦合
- 避免超长函数与过深嵌套
- 公共函数和类应写简短 docstring
- 仅在必要处添加注释，注释解释“为什么”，不是“代码表面在做什么”

## 状态与配置

- 配置读写逻辑集中管理
- 敏感信息不要硬编码进代码仓库
- 当前项目本地配置使用 `~/.config/open-router-key-viewer/config.json`
- 默认不自动保存 key，只有用户主动点击保存时才写入配置
- 提交时避免纳入真实密钥、令牌、用户隐私数据
- 避免把任何个人环境信息写入仓库，包括本机绝对路径、用户名、邮箱、主机名、仅对当前机器成立的目录结构、临时环境变量值和本地命令输出
- 新增文档、脚本、skill、注释或示例时，优先使用相对路径、通用占位符和与机器无关的表述

## 测试

- 默认使用 `pytest`
- 当前仓库还没有完善测试，新增业务逻辑时优先补单元测试
- UI 若不便做完整自动化，至少保证核心业务层可测试
- 修 bug 时，优先先补回归测试再修复

建议命令：

```bash
uv run pytest
```

## 开发流程

- 修改前先理解现有结构和调用链
- 优先修复根因，不做表面补丁
- 若引入新依赖，说明原因并保持最小化
- 对较大改动，拆成清晰的提交单元

## 提交要求

- 提交信息应清楚描述意图
- 功能改动、重构、修复尽量分开提交
- 不要把无关格式化、重命名和业务修改混在一起

## 对代理/协作者的要求

- 实现功能时默认遵循本文件
- 如仓库后续出现更具体的项目结构，以实际代码为准
- 若需求与本文件冲突，以用户明确要求为最高优先级
- 不确定时，优先选择更保守、更易维护的方案
